The Agent Identity Boundary

Browser agents and tool-using AI systems make a simple management question newly urgent: when work is done inside a real account, can the company prove whether the actor was a person, a software agent, or a delegated system acting under narrow authority?

The next enterprise AI failure may not look like a hallucination. It may look like a normal authenticated action. A calendar is changed. A supplier record is updated. A draft is sent. A support refund is approved. A file is read, summarised, and copied into a new workflow. Nothing appears exotic because the work happens through ordinary tools.

That is the point. As AI agents move from chat windows into browsers, APIs, internal tools, and authenticated sessions, the strategic risk shifts from answer quality to authority. The company has to know who or what acted, under whose permission, through which tool, for what purpose, and how the action can be challenged later.

NIST has made this direction explicit in 2026. Its AI Agent Standards Initiative points toward secure adoption, interoperability, agent authentication, identity infrastructure, and evaluations for human-agent and multi-agent interactions. Its May 2026 analysis of responses on AI agent security found broad agreement that agents create adoption-blocking security concerns and that ordinary cybersecurity practices still matter but need adaptation. OWASP's Agentic AI guidance says the same thing in practitioner language: autonomous agents create risks around tool misuse, context isolation, excessive agency, and indirect prompt injection.

Chrome's June 2026 WebMCP security guidance adds a practical browser example. Agents that use browser-exposed tools may operate inside a user's authenticated session. Tool definitions and tool outputs can carry hostile instructions. The model cannot be treated as a reliable boundary by itself. Once an agent can act in a logged-in context, identity and authority stop being IT plumbing. They become management controls.

An AI agent should never inherit a human identity wholesale. It should receive a bounded authority, a visible session, a narrow toolset, and a record that survives the work.

The Problem With Borrowed Identity

Most business software assumes a user is a person. The account carries permissions, history, approvals, audit trails, and blame. That assumption breaks when a person delegates work to an agent that can read pages, call tools, draft messages, update systems, or coordinate with other agents.

If the agent acts entirely as the user, the audit trail becomes misleading. The log says Maria changed the customer record, but the useful fact may be that Maria approved an agent to reconcile customer records from a source list, and the agent changed the field after reading an untrusted note. The difference matters for review, incident response, compliance, and trust. It also matters for ordinary management: a team cannot improve a workflow if it cannot tell where human judgment ended and delegated execution began.

The borrowed-identity problem is most dangerous in the browser because the browser is where business life already happens. Email, CRM, finance, HR, analytics, procurement, support, cloud consoles, and document systems all sit behind sessions that were designed for human navigation. An agent with a browser can appear low-friction precisely because it uses the same surfaces the employee uses. Low friction is also how authority leaks.

The Identity Boundary Ledger

A useful agent identity system does not start with a perfect standard. It starts with a ledger that product, security, legal, operations, and business owners can read. The ledger has seven lines.

Ledger Line Question Failure Mode
Actor Is the action performed by a person, a named agent, a background service, or a chain of agents? Every action appears to come from the human account.
Authority Source Who granted the agent permission, for which business purpose, and under which policy? Consent is implied by login rather than explicitly delegated.
Session When did authority begin, when does it expire, and what context is carried between tasks? A temporary task becomes a standing shadow employee.
Tool Scope Which tools, data, domains, actions, and write privileges are allowed? The agent can read broadly and act wherever the user can act.
Human Checkpoint Which decisions require review before execution, not after damage control? Approval is moved downstream into audit theatre.
Record Can the company reconstruct prompts, tool calls, data sources, approvals, outputs, and final actions? The company has a transcript but not an accountable operating record.
Revocation How is authority paused, narrowed, expired, or killed after a bad signal? The only stop rule is disabling the user's account or the whole system.

This is not a request for bureaucracy before experimentation. It is a way to let experimentation proceed without pretending that identity is solved by a checkbox. The smaller the agent, the lighter the ledger can be. But the ledger should exist before the agent can write, spend, message, approve, delete, transfer, or change production state.

The Browser Raises The Stakes

A browser agent is powerful because it can operate across systems without waiting for every vendor to build a native integration. It is risky for the same reason. The browser page contains trusted controls, untrusted content, third-party comments, hidden instructions, ads, documents, user-generated text, and tool metadata. An agent that cannot keep those contexts separate can be manipulated by the environment it is reading.

Chrome's WebMCP guidance names two concrete attack surfaces: malicious manifests and contaminated outputs. A tool description can include hostile instructions. A response from a site can contain content that tells the model to ignore previous instructions or misuse a tool. This is indirect prompt injection in the place where the agent is supposed to be useful: live, messy, authenticated work.

The management lesson is blunt. Do not treat the model as the control boundary. The boundary belongs around identity, authority, tools, confirmations, logs, and revocation. The model may reason well, but it should not be asked to be the only thing standing between an untrusted page and a privileged action.

Why Procurement Will Ask For It

Agent identity will become a procurement issue because buyers will not want to inherit another vendor's ambiguity. They will ask whether the system supports separate agent identities, delegated authorization, scoped credentials, tool-call logs, approval gates, audit exports, and emergency revocation. They will ask whether an action can be attributed to a human, an agent, a model, a tool, and a session. They will ask what happens when an agent uses a browser extension, iframe, API connector, workflow tool, or internal automation runner.

The companies selling agentic systems should welcome those questions. A vendor that can prove bounded authority has a commercial advantage over a vendor that says the customer can simply trust the user account. The latter may be easier to demo. The former is easier to approve.

There is a parallel with cloud. The cloud did not become enterprise infrastructure because every developer received broad production access. It became manageable when identity, roles, permissions, logs, secrets, and deployment controls became part of the operating model. AI agents will need the same managerial translation.

The Founder Version

A small company does not need a standards department. It needs a short rule: no production agent without a named owner, scoped authority, visible logs, and a kill switch. The first implementation can be plain language.

  1. Name the agent separately from the person who requested the work.
  2. Give it only the tools needed for the task.
  3. Keep write, spend, send, delete, and approve actions behind human confirmation until evidence says otherwise.
  4. Record the source material, tool calls, approvals, and final action.
  5. Expire the authority by default.

That discipline is enough to prevent the worst early mistake: letting a successful demo become a persistent actor inside the business without a clean boundary.

The Executive Test

Before any AI agent receives access to a browser session, internal tool, or production workflow, require eight answers.

  1. What is the agent's separate identity in logs and user-facing records?
  2. Who granted authority, and for which business purpose?
  3. Which data can it read, and which data is explicitly out of scope?
  4. Which tools can it call, and which actions require confirmation?
  5. How does the system handle malicious instructions from pages, files, comments, emails, tool descriptions, or tool outputs?
  6. What evidence proves the difference between human judgment, agent reasoning, tool execution, and final approval?
  7. When does the authority expire?
  8. How can the company pause, revoke, investigate, and repair after a bad action?

If those answers are missing, the company is not deploying an agent. It is lending a human identity to software and hoping the receipt never matters.

The strategic opportunity is real. Agents can compress tedious work, coordinate across systems, and make software feel less fragmented. But the more useful the agent becomes, the less acceptable borrowed identity becomes. The firms that solve the identity boundary early will be able to delegate more work with less institutional anxiety. The firms that skip it will discover, after the fact, that the most expensive question in agentic AI is not what the model knew. It is who was allowed to act.
